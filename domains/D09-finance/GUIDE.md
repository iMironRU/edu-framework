# D09 · Руководство по работе с доменом

## Как определить, что изменение относится к D09

Любое действие, которое влечёт **возникновение или изменение финансового обязательства
между ОО и обучающимся (заказчиком)**, — это D09.

| Ситуация | Домен |
|---|---|
| Студент подписывает договор на платное обучение | D09 |
| Бухгалтерия получает оплату от студента | D09 |
| Студент уходит в академотпуск | D05 (событие) + D09 (реакция: заморозка начислений) |
| Студент отчисляется — считается возврат | D09 |
| Стоимость обучения индексируется | D09 |
| Приказ об отчислении издаётся | D05 (событие) |
| ОО контролирует наполняемость группы | D04 |
| Студент переводится с платного на бюджет | D05 (приказ) + D09 (расторжение договора) |

## Принцип план / факт

D09 строится на жёстком разделении:

- **План** — договор, версия плана, график по периодам. Может версионироваться (допсоглашение,
  индексация, приказ). Говорит: «ожидается».
- **Факт** — начисление, платёж, возврат. Неизменяем после возникновения. Говорит: «случилось».

Никогда не правьте факты напрямую. Ошибка в факте гасится сторно-записью со ссылкой на
исходный факт; нетто вычисляется. Исправление через удаление — нарушение аудитоспособности.

## Откуда берутся начисления

Начисление не создаётся вручную и не появляется по расписанию. Оно возникает **по
академическому событию** из D05:

| Событие D05 | Действие в D09 |
|---|---|
| Зачисление | Первое начисление (если дата начала в новом периоде) |
| Перевод на следующий период | Начисление за новый период |
| Оставление на повторное обучение | Новая версия плана + начисление |
| Академотпуск | Заморозка: новые начисления не порождаются |
| Возврат из академотпуска | Возобновление начислений |
| Отчисление | Сторно незавершённого периода; расчёт возврата |

Дата события — **дата вступления в силу** (не дата издания приказа). Приказы часто
оформляются задним числом; финансовый контур должен это обрабатывать через сторно и
повторный расчёт.

## Жизненный цикл договора

```
Проект → Заключён → Действует ← → Приостановлен
    ↓         ↓            ↓
Аннулирован         Расторгнут → [финансово открыт до сальдо=0] → Исполнен
```

- **Проект** и **Приостановлен** — новые начисления не порождаются.
- **Расторгнут** — академически закрыт, но финансово может оставаться открытым (взыскание
  долга или оформление возврата).
- **Финансовое закрытие** — отдельное событие при сальдо = 0.

## Типовые ошибки

**Прямое редактирование начисления вместо сторно.**
Нарушает неизменяемость факта. Аудитор увидит не историю, а «послеправленные» цифры. Всегда
используйте: сторно исходного начисления + новое начисление на правильную сумму.

**Возврат средств оформлен как сторно платежа.**
Сторно платежа — исправление ошибки учёта (например, дубль). Возврат денег клиенту —
самостоятельное движение средств (отдельная запись). Путаница ведёт к некорректному балансу.

**Цена введена вручную, а не из прейскуранта.**
Полная стоимость договора — не отдельный реквизит, а сумма периодных цен из прейскуранта.
Ввод «от руки» создаёт расхождение с НСИ.

**Зачисление до заключения договора.**
Договор — предусловие зачисления. Зачисление без договора невозможно даже если оплата ещё
не поступила (договор может быть ещё не оплачен, но должен быть заключён).

**Деятельностная единица возврата без данных журнала.**
Если единица расчёта — занятия или ЗЕ, а журнальных данных нет, расчёт должен
блокироваться с сигналом, а не производиться по неполным данным или «на глаз».

**Договор закрыт сразу после приказа об отчислении.**
Академическое расторжение ≠ финансовое закрытие. Пока сальдо ≠ 0, договор финансово
открыт. Закрыть досрочно — потерять возможность контролировать взыскание долга или возврат.

## Что делать при смене учётной политики

Учётная политика (единица расчёта возврата, FIFO-правила, льготные категории) задаётся
каскадом: услуга → вид образования → организация. При смене:

1. Новое значение вступает в силу с даты нового НПА или локального приказа.
2. Периоды, по которым триггер уже сработал (начисления уже возникли), не пересчитываются.
3. Изменение применяется только к будущим начислениям и будущим договорам.

## Периодичность контрольных действий

| Действие | Периодичность |
|---|---|
| Сверка начислений с приказами D05 (нет «висящих» без события) | Ежемесячно |
| Проверка сальдо расторгнутых договоров | Ежемесячно |
| Контроль просроченных дебиторских задолженностей (D09-P06) | Ежемесячно |
| Мониторинг ожидаемых поступлений (маткапитал, кредит) | Ежеквартально |
| Проверка актуальности ПП № 1441 и его преемника | До 31.12.2026 и при выходе НПА |

## Интеграционная граница с D05

D09 **потребляет** события из D05, но не создаёт их. Если финансовый контур обнаруживает
несоответствие (начисление без события, событие без договора) — это сигнал о проблеме в
D05, а не повод порождать «компенсирующие» записи в D09.

Правило: финансовый контур доверяет идемпотентным событиям D05 (один приказ = одно событие),
но не доверяет им вслепую — каждое событие валидируется перед реакцией.
