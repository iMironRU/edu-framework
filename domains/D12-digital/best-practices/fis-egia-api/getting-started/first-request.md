# Первый запрос

Пошаговый гайд от нуля до успешного ответа. Используем `/api/cls/get` — самый простой запрос, не требующий подписи данных.

## Предварительные требования

- [ ] Есть ОГРН и КПП организации
- [ ] Сертификат загружен в ЛК ФИС
- [ ] Известен адрес тестового сервера
- [ ] Есть инструмент для работы с ЭП (КриптоПро или аналог)

## Шаг 1. Получить Session-Key

Сформируй Token-Header:

```json
{
  "Ogrn": "ВАШ_ОГРН",
  "Kpp": "ВАШ_КПП",
  "Month": 5
}
```

Закодируй в Base64:

```bash
echo -n '{"Ogrn":"ВАШ_ОГРН","Kpp":"ВАШ_КПП","Month":5}' | base64
```

Подпиши значение Token-Header своим сертификатом (PKCS#7 detached), получи `signature_base64`.

Отправь запрос:

```bash
curl -X POST https://АДРЕС_СЕРВЕРА/api/session/new \
  -H "Content-Type: application/json" \
  -H "Token-Header: ТВОЙ_BASE64_HEADER" \
  -d '{"signature_base64":"ТВОЯ_ПОДПИСЬ"}'
```

Ожидаемый ответ:

```json
{
  "Session-Key": "skey+...+2025-05-12_23-59"
}
```

Сохрани `Session-Key` — он нужен для всех следующих запросов.

## Шаг 2. Получить классификатор GenderCls

Это синхронный запрос, подпись не нужна.

```bash
curl -X GET "https://АДРЕС_СЕРВЕРА/api/cls/get?ClsType=GenderCls" \
  -H "Session-Key: ВАШ_SESSION_KEY"
```

Ожидаемый ответ — список полов с идентификаторами. Запомни значения `Id` — они используются в поле `IdGender` при создании поступающего.

## Шаг 3. Убедиться что всё работает

Если получил классификатор — интеграция работает на базовом уровне. Следующий шаг:

## Частые проблемы при первом запуске

**401 при `/api/session/new`**  
Сертификат не найден в ФИС. Проверь, загружен ли сертификат в ЛК, и совпадают ли ОГРН/КПП.

**400 с "Ошибка кодировки"**  
Проверь Base64-кодировку Token-Header — не должно быть переносов строк (`base64 -w 0` в Linux).

**Подпись не проходит**  
Убедись что подписываешь именно строку из Token-Header (не JSON-объект, а уже Base64-строку).
